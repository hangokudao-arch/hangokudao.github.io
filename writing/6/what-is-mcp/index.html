MCP는 무엇인가: AI에게 도구를 연결하는 표준

MCP를 처음 듣는 사람을 위해 Model Context Protocol이 왜 나왔고, host, client, server, tool, resource를 어떻게 이해하면 되는지 초보자 눈높이로 정리한다.

이 글에서 다루는 것

  • MCP는 AI 앱이 외부 도구와 자료를 공통 방식으로 연결하기 위한 표준이다.
  • 초보자는 host, client, server, tool, resource만 먼저 구분해도 MCP 흐름을 따라갈 수 있다.
  • MCP는 편리하지만 보안 장치가 아니므로 trusted server, 승인, 권한 범위, 로그 검토를 같이 봐야 한다.

앞 글에서는 Skill을 반복 작업 절차로 봤다. 그 글 끝에서 MCP라는 말을 잠깐 지나갔다.

처음 보면 MCP도 Skill, plugin, connector, API 같은 말과 섞여 보인다. 그래서 더 어렵게 느껴진다. 하지만 처음부터 protocol 세부 구현을 볼 필요는 없다.

나는 MCP를 이렇게 이해하는 편이 가장 쉬웠다.

AI가 혼자 알고 있는 것만으로 답하지 않고, 필요할 때 외부 도구와 자료에 정해진 방식으로 연결되는 통로.

이 글은 MCP를 처음 듣는 사람을 위한 정리다. 서버를 직접 만드는 법보다, 왜 필요한지, 어떤 단어만 먼저 알면 되는지, 어디서 조심해야 하는지를 본다.

AI는 기본적으로 내 파일과 앱을 모른다

AI에게 “내 캘린더 보고 내일 일정 정리해줘”라고 말해도, AI가 자동으로 내 캘린더를 볼 수 있는 것은 아니다.

“이 저장소의 issue를 보고 우선순위를 정리해줘”라고 말해도, AI가 자동으로 GitHub 권한을 갖는 것도 아니다. 모델은 대화에 들어온 텍스트와 연결된 도구가 돌려준 결과를 보고 판단한다.

그래서 AI 앱에는 연결 통로가 필요하다.

예전 방식은 서비스마다 따로 붙이는 쪽에 가까웠다. 어떤 AI 앱은 Google Drive 연결을 따로 만들고, 어떤 앱은 GitHub 연결을 따로 만들고, 또 다른 앱은 내부 DB 연결을 따로 만든다. 이렇게 되면 같은 일을 하려 해도 앱마다 연결 방식과 권한 관리가 달라진다.

MCP(Model Context Protocol)는 이 문제를 줄이기 위해 나온 표준이다. 공식 MCP 문서는 MCP를 AI 애플리케이션에 context를 제공하는 공개 표준으로 설명한다. 여기서 context는 모델이 판단할 때 볼 수 있는 자료, 도구, 상태를 뜻한다.

쉽게 말하면 MCP는 “AI 앱과 외부 도구가 서로 알아들을 수 있게 맞춘 약속”이다.

비유하면 충전기 규격에 가깝다

MCP를 처음 설명할 때 나는 충전기 규격 비유가 꽤 도움이 된다고 본다.

예전에는 기기마다 충전 단자가 달랐다. 충전기는 많지만 서로 맞지 않았다. 표준 단자가 생기면 모든 문제가 사라지는 것은 아니지만, 최소한 “어떻게 꽂을지”부터 매번 새로 만들 필요는 줄어든다.

MCP도 비슷하다.

AI 앱은 “어떤 도구가 있니”, “이 도구는 어떤 입력을 받니”, “이 자료를 읽을 수 있니”라고 물을 수 있다. MCP server는 “나는 이런 tool과 resource를 제공한다”라고 답한다. 둘 사이의 약속이 맞으면 AI 앱은 그 도구를 사용할 후보로 볼 수 있다.

이 비유에서 중요한 점은 표준이 곧 안전을 뜻하지 않는다는 것이다.

충전기 단자가 맞아도 불량 충전기를 꽂으면 위험하다. MCP도 마찬가지다. 연결 방식이 표준화됐다는 말은 아무 server나 믿어도 된다는 뜻이 아니다.

먼저 알아둘 단어 다섯 개

MCP를 처음 볼 때는 다섯 단어만 먼저 보면 된다.

첫째, host다.

Host는 사용자가 실제로 쓰는 AI 앱이다. 예를 들어 데스크톱 AI 앱, 코딩 에이전트, IDE 확장, ChatGPT 같은 표면이 여기에 가깝다. 사용자의 요청을 받고, 어떤 MCP server를 연결할지 관리한다.

둘째, client다.

Client는 host 안에서 MCP server와 말하는 작은 연결 담당이라고 보면 된다. 사용자가 직접 client를 볼 일은 많지 않다. 중요한 것은 host가 server와 직접 아무 말이나 하는 것이 아니라, client를 통해 정해진 MCP 방식으로 주고받는다는 점이다.

셋째, server다.

Server는 외부 도구나 자료를 MCP 방식으로 내보내는 쪽이다. 예를 들어 파일 검색, 일정 조회, GitHub issue 확인, 데이터베이스 조회 같은 기능을 server가 제공할 수 있다.

넷째, tool이다.

Tool은 AI가 실행할 수 있는 동작이다. “파일을 검색한다”, “issue를 읽는다”, “캘린더 일정을 찾는다”, “테스트를 실행한다” 같은 일이 tool에 가깝다. Tool은 대개 입력을 받고 결과를 돌려준다.

다섯째, resource다.

Resource는 AI가 읽을 수 있는 자료다. 문서, 파일 내용, 스키마, 설정 정보처럼 “실행”보다 “읽기”에 가까운 context다.

이렇게 보면 MCP 흐름은 단순해진다.

사용자 요청은 AI 앱(host)으로 들어간다. Host 안의 MCP client가 MCP server와 말하고, server는 tool을 실행하거나 resource를 돌려준다. 그 결과가 다시 AI의 context로 들어온다.

초보자에게는 이 정도면 충분하다. transport, JSON-RPC, schema 같은 말은 나중에 server를 직접 만들 때 봐도 늦지 않다.

MCP가 있으면 무엇이 달라질까

MCP가 필요한 이유는 AI가 더 “똑똑해져서”가 아니다. AI가 봐야 할 현재 자료와 실행할 도구를 더 일관된 방식으로 연결하기 위해서다.

예를 들어 AI가 내 프로젝트 문서를 읽고 답하게 만들고 싶다고 해보자. 그냥 프롬프트에 문서를 붙여 넣을 수도 있다. 하지만 문서가 많아지면 매번 붙여 넣을 수 없다. MCP server가 문서 검색 resource를 제공하면, AI 앱은 필요한 문서를 찾아 context로 가져올 수 있다.

또 다른 예로 GitHub issue를 정리한다고 해보자. AI에게 issue 목록을 복사해서 붙이면 한 번은 된다. 하지만 issue가 바뀌면 다시 복사해야 한다. MCP server가 issue 조회 tool을 제공하면, AI는 현재 issue를 읽고 요약할 수 있다.

OpenAI API 문서도 connectors와 remote MCP servers를 외부 서비스에 연결하고 필요한 경우 모델이 도구를 사용할 수 있게 하는 방식으로 설명한다. 이때 도구 호출은 자동으로 허용할 수도 있고, 개발자가 명시 승인 흐름을 요구하게 만들 수도 있다.

여기서 핵심은 “현재성”과 “행동 가능성”이다.

  • 현재성: 모델이 학습 때 알던 정보가 아니라 지금 server가 돌려준 자료를 볼 수 있다.
  • 행동 가능성: 단순 답변이 아니라 정해진 tool을 호출해 조회나 작업을 할 수 있다.
  • 재사용성: 앱마다 새 통합을 만들기보다 MCP 방식으로 server를 연결할 수 있다.

그래서 MCP는 AI agent가 실제 작업 환경에 들어올수록 중요해진다. 프롬프트만으로는 현재 파일, 현재 일정, 현재 issue, 현재 DB 상태를 안정적으로 다루기 어렵기 때문이다.

MCP는 Skill이나 AGENTS.md와 다르다

이 블로그의 앞 글들을 읽었다면 AGENTS.md, Skill, MCP가 다 비슷하게 보일 수 있다. 모두 AI가 더 잘 일하게 돕는 장치처럼 보이기 때문이다.

하지만 역할은 다르다.

AGENTS.md는 repo에서 항상 지켜야 할 기본 지시다. 예를 들어 “build 산출물은 직접 수정하지 않는다”, “push와 deploy는 확인 후 진행한다” 같은 기준을 둔다.

Skill은 특정 반복 작업의 절차다. 예를 들어 “발행 전 안전성을 점검한다”, “git 변경을 분리해서 commit한다”처럼 작업 종류가 맞을 때 꺼내 쓰는 workflow에 가깝다.

MCP는 외부 context와 tool을 연결하는 통로다. AI가 혼자 갖고 있지 않은 자료를 읽거나, 외부 서비스의 기능을 호출해야 할 때 쓴다.

나는 이렇게 나눠서 기억한다.

  • 프롬프트: 이번 요청의 목표
  • AGENTS.md: repo의 기본 운영 기준
  • Skill: 반복 작업 절차
  • MCP: 외부 도구와 자료를 연결하는 표준 통로
  • Hook, permission, CI: 실제로 막거나 검증하는 실행 경계

이 구분을 해두면 MCP를 과하게 기대하지 않게 된다. MCP server를 붙인다고 글을 잘 쓰는 기준이 생기지는 않는다. 그건 AGENTS.md나 Skill, 검수 기준 쪽 문제다. 반대로 AGENTS.md를 아무리 잘 써도 AI가 내 캘린더나 GitHub issue를 자동으로 볼 수는 없다. 그건 connector나 MCP server가 필요한 영역이다.

가장 조심할 것은 보안이다

MCP는 편리하지만, 초보자가 가장 먼저 봐야 할 것은 보안이다.

MCP server는 AI에게 tool과 resource를 제공한다. 이 말은 server가 모델의 context에 들어온 정보를 받을 수 있고, 어떤 경우에는 외부 서비스에서 행동을 할 수도 있다는 뜻이다.

OpenAI 문서는 remote MCP server를 신뢰할 수 있어야 하며, 악의적인 server가 모델 context에 들어온 민감한 정보를 빼낼 수 있다고 경고한다. MCP 공식 보안 가이드도 authorization, token handling, prompt injection, tool permission 같은 항목을 별도로 다룬다. 연결이 쉬워진 만큼, 설계와 운영에서 보안 기준을 같이 잡아야 한다는 뜻이다.

초보자는 아래 네 가지를 먼저 기억하면 된다.

  • 공식 또는 신뢰할 수 있는 server부터 쓴다.
  • 처음에는 읽기 권한만 주고, 쓰기나 삭제 tool은 나중에 연다.
  • 민감한 tool은 승인 없이 실행되지 않게 한다.
  • 어떤 데이터가 server로 나가는지 로그와 화면에서 확인한다.

특히 “승인 없이 실행”은 조심해야 한다. 캘린더를 읽는 것과 일정을 삭제하는 것은 위험도가 다르다. 문서를 검색하는 것과 외부로 전송하는 것도 다르다. MCP를 붙일 때는 “이 server가 무엇을 할 수 있는가”를 먼저 봐야 한다.

나는 MCP를 켤 때 아래 질문을 먼저 던지는 편이 낫다고 본다.

  • 이 server가 읽을 수 있는 것은 무엇인가?
  • 이 server가 바꿀 수 있는 것은 무엇인가?
  • 내가 승인해야 하는 action은 무엇인가?
  • 실수했을 때 되돌릴 수 있는가?

이 질문에 답하지 못하면 아직 자동 실행으로 넘길 단계가 아니다.

언제 MCP를 써볼 만한가

MCP가 항상 필요한 것은 아니다.

한 번 읽고 끝낼 짧은 문서는 프롬프트에 붙여 넣는 편이 더 단순하다. repo 안의 반복 규칙은 AGENTS.md가 더 낫다. 자주 반복하는 작업 순서는 Skill이 더 맞을 수 있다.

MCP는 외부의 현재 상태가 필요할 때 쓸 만하다.

  • 문서 저장소에서 최신 문서를 찾아야 한다.
  • GitHub issue, PR, discussion 같은 현재 상태를 읽어야 한다.
  • 캘린더, 메일, 파일 저장소처럼 계정 권한이 붙은 자료가 필요하다.
  • 데이터베이스나 내부 API처럼 매번 바뀌는 값을 조회해야 한다.
  • 단순 조회를 넘어 정해진 tool을 실행해야 한다.

반대로 아래 상황이면 기다리는 편이 낫다.

  • 어떤 권한을 주는지 이해하지 못했다.
  • server 제공자를 신뢰할 수 없다.
  • 삭제, 결제, 배포, 권한 변경 같은 되돌리기 어려운 tool이 승인 없이 열린다.
  • 민감한 고객 데이터나 private 문서가 어디로 가는지 추적할 수 없다.

처음에는 “읽기 전용 + 공식 server + 승인 필요” 정도가 좋은 출발점이다.

오늘 가져갈 기준

MCP를 한 문장으로 줄이면 이렇다.

MCP는 AI 앱이 외부 도구와 자료를 공통 방식으로 연결하기 위한 표준이다.

하지만 이 한 문장만으로는 부족하다. 실제 작업에서는 다음 기준이 더 중요하다.

  • MCP는 모델이 아니다. 외부 context와 tool을 연결하는 통로다.
  • MCP는 보안 장치가 아니다. 권한, 승인, 로그, 신뢰할 server가 따로 필요하다.
  • MCP는 Skill이나 AGENTS.md를 대체하지 않는다. 반복 기준과 외부 연결은 역할이 다르다.
  • MCP server를 붙이면 모델이 볼 수 있는 자료와 할 수 있는 행동이 늘어난다. 그래서 편리함과 위험이 같이 늘어난다.
  • 초보자는 host, client, server, tool, resource만 먼저 구분하면 된다.

나는 MCP를 “AI에게 손과 눈을 붙이는 방식”으로만 설명하는 것을 조심한다. 그 비유는 쉽지만, 권한과 책임을 흐릴 수 있다. 더 정확히는 AI 앱이 외부 세상과 말하기 위한 표준 통로다. 통로를 열면 무엇이 들어오고 나가는지 봐야 한다.

다음에 MCP를 실제로 붙일 일이 생기면, 기능 목록보다 먼저 권한 목록을 보려고 한다. 어떤 tool이 있는지보다, 그 tool이 무엇을 읽고 무엇을 바꿀 수 있는지가 먼저다.

그 정도로 시작하면 충분하다.
