Codex Skill은 무엇인가: AGENTS.md와 어떻게 다를까

Codex Skill을 SKILL.md 중심의 재사용 절차 묶음으로 설명하고, AGENTS.md, 실행 검사(hook), 외부 도구 연결(MCP), 설치 묶음(plugin)과 어떻게 나눠야 하는지 정리한다.

  • #codex
  • #agent-skills
  • #agents-md
  • #workflow

이 글에서 다루는 것

  • Skill은 SKILL.md를 중심으로 반복 작업 절차를 재사용하게 만드는 작은 묶음이다.
  • AGENTS.md는 repo 기본값, Skill은 특정 반복 작업 절차, hook과 CI는 실행 경계로 나눠 본다.
  • Skill을 많이 설치하기보다 내가 설명할 수 있는 절차만 작게 남기는 기준을 제안한다.

앞 글에서는 반복 지시를 컨텍스트로 옮겨야 한다고 정리했다.

그다음 질문은 이렇다.

AGENTS.md까지 만들었다면, Codex Skill은 왜 필요한가?

답은 AGENTS.md가 부족해서가 아니다. 역할이 다르기 때문이다. AGENTS.md는 repo에서 항상 지켜야 할 기본값을 둔다. 새 글은 draft: true로 시작한다, dist/를 직접 수정하지 않는다, push나 deploy는 확인 전 하지 않는다는 식이다.

Skill은 더 좁다. 특정 작업을 반복할 때 다시 쓰는 절차 묶음이다. 예를 들어 “Codex 기능 글을 쓸 때 공식 문서와 개인 경험을 분리한다”, “commit 전 변경 파일을 분리해서 본다”, “UI 작업 전 모바일과 데스크톱 확인 기준을 잡는다” 같은 일에 맞다.

이 글은 Skill을 SKILL.md 중심의 재사용 절차로 설명하고, 어떤 기준은 프롬프트(prompt), AGENTS.md, 실행 검사(hook), CI, 외부 도구 연결(MCP), 설치 묶음(plugin) 쪽에 남겨야 하는지 나눠 본다.

핵심은 모든 반복 기준을 한곳에 모으지 않는 것이다. 이번 한 번이면 프롬프트(prompt), repo 전체 기본값이면 AGENTS.md, 특정 작업에서 반복되는 절차면 Skill, 반드시 막아야 하면 권한(permission), CI, branch protection, hook 같은 실행 장치를 본다.

반복 비용이 Skill 후보가 된다

Skill 후보는 그럴듯한 주제에서 나오지 않는다. 같은 일을 세 번쯤 반복했는데 매번 빠지는 단계가 보이면 그때 후보가 된다.

예를 들어 “공식 문서 확인”, “변경 파일 분리”, “모바일 화면 확인”, “발행 전 공개 안전성 점검”은 한 번의 취향이 아니다. 특정 작업을 할 때 반복해서 필요한 절차다. 이런 절차를 매번 프롬프트에 붙이면 언젠가 빠진다.

여기서 바로 Skill을 만들 필요는 없다. 먼저 “이번 한 번의 요청인가, repo 전체 기본값인가, 특정 작업에서만 반복되는 절차인가”를 나눠보면 된다. 이 구분이 잡히면 프롬프트에는 이번 요청의 목표가 남고, AGENTS.md에는 repo 기본값이 남고, Skill에는 특정 작업을 다시 할 때 필요한 순서가 남는다.

내가 이 구분을 신경 쓰게 된 이유도 비슷하다. 처음에는 “공식 문서 확인”, “git 변경 분리”, “블로그 공개 안전성 점검”을 전부 프롬프트에 붙였다. 그러다 보니 어떤 기준은 repo 전체 규칙처럼 굳어졌고, 어떤 기준은 Codex 글을 쓸 때만 필요했고, 어떤 기준은 실제 차단 장치가 필요한 위험 작업이었다. 이 셋을 같은 칸에 넣으면 AGENTS.md도 길어지고, Skill도 만능 체크리스트처럼 부풀었다.

공식 문서에서 말하는 Skill

이 정의가 필요했던 이유는 거창한 기능 이름을 외우기 위해서가 아니다. 내가 매번 프롬프트에 붙이던 반복 절차를 어디에 둘지 정해야 했기 때문이다.

OpenAI Codex의 Agent Skills 문서는 Skill을 반복 가능한 workflow를 위한 재사용 capability로 설명한다. 보통 SKILL.md가 중심이고, 필요하면 scripts/, references/, assets/ 같은 보조 파일을 함께 둔다.

중요한 점은 Skill이 “프롬프트 묶음”이 아니라는 것이다. Skill은 에이전트가 특정 작업을 할 때 어떤 순서로 판단하고, 어떤 자료를 읽고, 어떤 검증을 해야 하는지 묶어두는 작업 절차에 가깝다.

공식 문서에서 특히 중요한 점은 progressive disclosure다. 한국어로 풀면 “필요할 때만 자세히 읽기”에 가깝다. Codex는 먼저 Skill의 이름, 설명, 경로 같은 metadata를 보고, 해당 Skill이 선택됐을 때 SKILL.md를 읽고, reference 문서나 script는 필요할 때만 본다.

내가 지금 이해하는 정의는 이렇다.

Skill은 SKILL.md를 중심으로 반복 작업 절차를 묶어두고,
필요할 때만 references, scripts, assets를 더 읽게 하는 작은 작업 절차 패키지(workflow package)다.

AGENTS.md와 Skill은 역할이 다르다

둘 다 에이전트에게 지시를 주는 파일처럼 보인다. 그래서 처음에는 헷갈린다.

나는 지금 이렇게 나눠서 본다.

  • 프롬프트(prompt)는 이번 요청의 목표와 예외다. “이 글은 초안까지만 만들어줘” 같은 말이 여기에 가깝다.
  • AGENTS.md는 repo에서 항상 지켜야 할 기본값이다. “새 글은 draft: true로 시작한다” 같은 기준은 매번 프롬프트에 붙일 일이 아니다.
  • Skill은 특정 작업을 반복할 때 쓰는 절차다. “OpenAI 공식 문서를 확인한 뒤 Codex 설명을 작성한다”처럼 작업 종류가 맞을 때만 꺼내 쓰는 쪽에 가깝다.
  • Permission, CI, branch protection은 실제 차단이나 공개 전 검증 장치다. 위험 명령 승인, push 차단, lint, type check처럼 문서가 아니라 실행 경계가 필요한 일에 맞다.
  • Hook은 특정 이벤트에서 실행되는 검사 장치다. 예를 들어 tool call 전 확인이나 작업 종료 시 검증처럼 정해진 시점에 붙는다.
  • MCP server는 외부 도구와 맥락을 연결하는 통로다. 문서 서버, GitHub, Figma처럼 현재 상태(live context)가 필요할 때 본다.
  • App이나 Connector는 인증된 앱 통합이다. 보통 설치 가능한 묶음(plugin)이 제공하며, 계정 기반 문서, 저장소, 업무 도구 연결처럼 권한이 붙은 외부 맥락에 가깝다.

내 블로그 작업으로 가져오면 더 단순하다. “이번 글은 짧게”는 프롬프트다. “새 글은 draft: true”는 AGENTS.md다. “Codex 글을 쓸 때 공식 문서와 개인 경험을 분리한다”는 Skill 후보다. “승인 전 push를 막는다”는 문서만으로 끝낼 일이 아니라 hook이나 permission까지 같이 볼 문제다.

AGENTS.md는 repo의 운영 기준이다. 작업 전 어디를 봐야 하는지, 어떤 명령으로 검증해야 하는지, 무엇을 하면 안 되는지처럼 반복되는 기본값을 둔다. Codex에서는 global, project, 현재 작업 경로에 가까운 instruction이 층을 이루기 때문에, 가까운 파일일수록 더 구체적인 기준이 될 수 있다.

Skill은 더 좁다. 작업 종류가 맞을 때만 꺼내 쓰는 workflow다. 공식 문서 확인, git 안전 점검, UI 디자인 기준, 긴 글 편집, 릴리스 점검처럼 “이 일을 할 때는 항상 이 순서가 필요하다” 싶은 것에 맞다.

그래서 Skill은 AGENTS.md를 대체하지 않는다. 오히려 AGENTS.md를 너무 길게 만들지 않기 위해 필요한 분리 단위에 가깝다.

Skill 후보는 세 가지 질문으로 걸러낸다

Skill로 만들 만한 일에는 공통점이 있다. 같은 작업을 여러 번 반복하고, 매번 빠뜨리면 안 되는 단계가 있으며, 참고해야 할 문서나 검증 명령이 있다. 한 번의 프롬프트보다 길지만 모든 작업에 항상 필요한 규칙은 아닐 때 Skill 후보가 된다.

나는 세 가지 질문으로 나눠 본다.

  • 오늘 한 번만 필요한가. 그렇다면 프롬프트에 둔다.
  • 이 repo에서 항상 지켜야 하는가. 그렇다면 AGENTS.md에 둔다.
  • 특정 작업에서 반복되는 절차인가. 그렇다면 Skill 후보로 본다.

예를 들어 “Codex 기능 설명을 블로그 글로 쓴다”는 작업은 Skill에 잘 맞는다. 공식 문서 확인, 개인 경험과 공식 사실 분리, 출처 표기, 공개 안전성 점검이 반복되기 때문이다.

반대로 “오늘은 리뷰만 하고 수정하지 마”는 프롬프트다. “이 repo에서는 dist/를 직접 수정하지 않는다”는 AGENTS.md다. “draft: false 변경이나 push를 실제로 막는다”는 Skill만으로는 부족하고, 권한(permission), CI, branch protection, hook 같은 실행 경계를 봐야 한다.

잘못 배치하면 비용이 생긴다. 한 번짜리 취향을 AGENTS.md에 넣으면 repo 기본값이 오염된다. 실제 차단이 필요한 일을 Skill에만 맡기면 문서상 기준은 있어도 실행 경계가 비어 있다. 외부 시스템 접근이 필요하지 않은데 MCP부터 붙이면 검토해야 할 권한과 실패 지점만 늘어난다.

늘릴 때는 설치보다 설명 가능성을 본다

처음에는 공개 repo에서 좋아 보이는 Skill을 많이 가져오면 에이전트가 더 잘 일할 것처럼 보인다. 실제로는 반대 문제가 먼저 생긴다.

어떤 Skill이 언제 불리는지 내가 설명하지 못하면, 결과가 좋아져도 왜 좋아졌는지 알 수 없다. 반대로 결과가 이상해졌을 때도 어떤 기준이 개입했는지 추적하기 어렵다.

OpenAI 문서 기준으로도 Skill 자동 호출은 name, description, 경로 같은 metadata에서 시작한다. Skill이 많거나 설명이 겹치면 어떤 절차가 작동했는지 사람이 따라가기 어려워진다. 많이 설치하는 것 자체가 곧 품질이 되지는 않는다.

가져오기 전에 최소한 아래는 봐야 한다.

  • 이 Skill은 어떤 상황에서 호출되어야 하는가.
  • description만 봐도 trigger와 제외 조건이 분명한가.
  • 내 repo의 AGENTS.md와 충돌하지 않는가.
  • 파일 삭제, push, deploy, credential 접근 같은 위험 작업을 직접 하라고 시키지 않는가.
  • script가 있다면 무엇을 실행하는지 이해할 수 있는가.

특히 script가 붙은 Skill은 더 보수적으로 봐야 한다. instruction-only Skill은 잘못돼도 대개 지시 품질 문제로 끝난다. 하지만 script는 실제 파일, 명령, 외부 시스템 접근으로 이어질 수 있다. 그래서 “좋아 보인다”보다 “무엇을 하는지 설명할 수 있다”가 먼저다.

직접 만들 때는 작게 시작한다

Skill을 직접 만들 때도 처음부터 크게 만들 필요가 없다.

처음에는 SKILL.md 하나면 충분하다.

---
name: blog-source-check
description: 블로그 글에서 공식 사실과 개인 경험을 분리해야 할 때 사용한다.
---

1. 글의 핵심 주장을 목록으로 뽑는다.
2. 각 주장을 공식 문서, 개인 경험, 추정으로 분류한다.
3. 공식 문서가 필요한 주장은 출처를 확인한다.
4. 확인하지 못한 문장은 단정하지 말고 "확인 필요"로 낮춘다.
5. 본문에는 공개하면 안 되는 내부 경로나 비공개 대화 원문을 남기지 않는다.

이 정도만 있어도 “이 글 사실성 좀 봐줘”라는 프롬프트보다 낫다. 해야 할 일이 단계로 남고, 다음에도 같은 방식으로 반복할 수 있기 때문이다.

다만 description은 더 구체적으로 다듬는 편이 좋다. 나쁜 description은 “블로그를 잘 쓰게 한다”처럼 넓다. 좋은 description은 “AI agent 글에서 공식 사실, 개인 경험, 추정을 분리해야 할 때 사용한다. 단순 오탈자 수정에는 사용하지 않는다”처럼 trigger와 제외 조건이 같이 보인다.

처음부터 scripts를 붙일 필요는 없다. 반복 실행해야 하는 검증 명령이 생기거나, 사람이 매번 똑같이 하는 기계적 작업이 보일 때 붙여도 늦지 않다.

Skill을 많이 갖는 것보다 중요한 건 내가 설명할 수 있는 Skill만 남기는 것이다. 에이전트가 똑똑해지는 느낌보다, 내가 검토할 수 있는 작업 절차가 늘어나는 쪽이 더 믿을 만하다.

결론

AGENTS.md를 만들었다고 Skill이 필요 없어지는 것은 아니다.

둘은 층이 다르다. AGENTS.md는 repo의 기본 운영 기준이다. Skill은 특정 반복 작업을 위한 재사용 절차다. 프롬프트(prompt)는 이번 요청의 목표다. 권한(permission), CI, branch protection은 더 강한 차단과 검증 장치다. Hook은 특정 이벤트에서 위험을 줄이는 실행 검사다. MCP server와 App(Connector)은 외부 시스템을 다루는 통로다.

이 층을 나누지 않으면 모든 것이 프롬프트로 밀려온다. 그러면 매번 길게 설명하고, 어느 날 한 줄을 빠뜨리고, 나중에는 에이전트가 왜 그렇게 움직였는지 설명하기 어려워진다.

다음 반복 작업을 만났을 때는 다섯 문장만 물어보면 된다.

  • 이건 오늘 한 번의 요청인가.
  • 이 repo에서 항상 지켜야 하는 기본값인가.
  • 특정 작업에서 세 번 이상 반복한 절차인가.
  • 실제로 막아야 하는 위험 작업인가.
  • 내가 읽고 설명할 수 있는 Skill인가.

Skill은 에이전트를 마법처럼 바꾸는 장치가 아니다. 같은 일을 할 때 같은 실수를 덜 하게 만드는 작은 작업 절차다.

내가 Skill을 크게 보지 않으려는 이유도 여기에 있다. 설명할 수 없는 자동화보다, 다음 작업에서 빠뜨리지 않을 작은 절차 하나가 더 오래 간다.

그 정도로 보면 충분하다.

다음 글에서는 Skill처럼 절차를 묶는 문제에서 한 걸음 더 나가, MCP가 AI에게 외부 도구와 자료를 연결하는 방식을 초보자 눈높이로 정리한다.
