에이전트 워크플로

같은 말을 세 번째 붙여넣고 있다면 규칙 파일로 빼야 한다

AI 코딩 에이전트에게 같은 금지 조건을 반복해서 붙여넣던 경험을 바탕으로, 프롬프트와 규칙 파일을 나누는 기준을 정리한다.

이 글에서 다루는 것

  • 같은 금지 조건을 반복해서 붙여넣는 순간을 규칙 파일로 옮겨야 할 신호로 본다.
  • 프롬프트에는 이번 작업의 부탁을 남기고, 반복되는 repo 기본값은 AGENTS.md나 CLAUDE.md로 분리한다.
  • 규칙 파일은 안전장치가 아니라 멈춤 조건과 확인 조건을 남기는 운영 기준이라는 점을 구분한다.

처음에는 프롬프트를 더 길게 쓰면 해결될 줄 알았다.

새 글은 draft: true로 시작해줘. dist/는 건드리지 마. push나 deploy는 묻고 해. 완료라고 말하기 전에 npm run check와 npm run build를 돌려줘.

이런 문장을 한 번 붙일 때는 별생각이 없었다. 두 번째까지도 그럴 수 있다고 봤다. 그런데 세 번째부터는 좀 이상했다. 내가 AI에게 일을 맡기는 게 아니라, 같은 주의사항을 매번 받아쓰기 시키고 있었다.

그때부터 프롬프트를 더 다듬기 전에 규칙 파일을 먼저 보게 됐다.

세 번째 반복은 신호다

프롬프트에 넣을 말과 규칙 파일에 둘 말은 다르다.

프롬프트는 이번 작업의 부탁에 가깝다.

이번 글은 공개하지 말고 초안까지만 만들어줘.
이 함수만 보고 버그 가능성을 확인해줘.
오늘은 코드 수정 없이 리뷰만 해줘.

이런 말은 그날의 요청이다. 다음 작업에서는 달라질 수 있다.

반대로 매번 다시 쓰는 문장이 있다.

빌드 산출물은 직접 수정하지 않는다.
새 글은 기본적으로 draft 상태로 시작한다.
삭제, push, deploy는 확인 전에는 하지 않는다.
완료 전 가능한 검증 명령을 실행한다.

이건 취향이 아니라 repo의 기본값에 가깝다. 이런 문장이 채팅창에 계속 남아 있으면, 프롬프트가 길어진 것이 아니라 운영 기준이 흩어진 것이다.

프롬프트가 나쁜 게 아니다

프롬프트는 여전히 필요하다.

문제는 프롬프트가 너무 많은 일을 떠안는 순간이다. 작업 목표, 출력 형식, 독자, 예외 조건까지는 프롬프트가 맡아도 된다. 그런데 수정 범위, 공개 여부, 검증 기준, 위험 작업 기준까지 매번 프롬프트에 넣기 시작하면 대화창이 작은 운영 매뉴얼이 된다.

처음에는 그게 편하다. 붙여넣으면 되니까.

그런데 세션이 바뀌면 다시 붙여야 한다. 한 줄이 빠지면 에이전트는 그 기준을 모르는 사람처럼 움직인다. 대화가 길어지면 중요한 기준과 그날의 취향이 섞인다. 나중에는 내가 무슨 일을 요청했는지보다, 무엇을 하지 말라고 했는지가 더 길어진다.

그 상태에서는 프롬프트를 더 잘 쓰는 것보다, 반복되는 말을 어디로 빼둘지 정하는 편이 낫다.

규칙 파일은 프롬프트 저장소가 아니다

AGENTS.md나 CLAUDE.md를 처음 보면 긴 프롬프트를 파일로 옮겨둔 것처럼 보인다. 나도 처음에는 그렇게 봤다.

그런데 실제로 쓰다 보면 역할이 조금 다르다. 이 파일들은 에이전트에게 성격을 부여하는 문서가 아니다. repo에서 일할 때 어디까지 보고, 어디서 멈추고, 무엇을 확인해야 하는지 남겨두는 작업 기준에 가깝다.

예를 들어 이런 문장은 규칙 파일에 더 잘 맞았다.

- 새 블로그 글은 `src/content/blog`에 둔다.
- 새 글은 기본적으로 `draft: true`로 시작한다.
- `dist/` 같은 build 산출물은 직접 수정하지 않는다.
- 삭제, push, deploy는 확인 전에는 하지 않는다.
- 완료 전 `npm run check`와 `npm run build`를 실행한다.

재미있는 문장은 아니다. 그래도 사고를 줄이는 문장은 대개 재미가 없다.

이런 기준이 repo 안에 있으면 다음 요청은 짧아진다. “이번 글은 실패담 톤으로 새 초안 하나 만들어줘”처럼 말할 수 있다. 글 위치, draft 기본값, 검증 명령, 공개 발행 금지는 파일이 이미 알고 있어야 한다.

그래도 막아주는 건 아니다

여기서 착각하면 안 되는 지점이 있다.

규칙 파일은 안전장치가 아니다. 에이전트가 참고할 context다. “push 전에 물어봐”라고 써두면 물어볼 가능성은 올라간다. 하지만 그것만으로 잘못된 push가 물리적으로 막히지는 않는다.

정말 막아야 하는 일은 별도 장치가 필요하다. Git hook(커밋이나 push 앞에서 실행되는 검사 스크립트), 권한, CI, branch protection, review 절차 같은 것들이다. 규칙 파일은 “여기서 멈춰야 한다”를 알려주는 쪽이고, hook이나 권한은 “멈추지 않으면 실패하게 만드는” 쪽이다.

나는 이 둘을 섞어 쓰면 글도 작업도 위험해진다고 본다. AGENTS.md가 있다고 해서 안전해졌다고 쓰면 과장이다. 더 정확한 말은 이 정도다.

반복 기준을 repo 안에 남겨서, 에이전트가 멈춰야 할 지점을 더 자주 보게 만든다.

딱 거기까지다. 그 이상은 다른 장치의 몫이다.

처음 버전은 짧아야 한다

규칙 파일을 처음 만들 때 욕심을 내기 쉽다. 좋은 코드, 좋은 UX, 깔끔한 설계, 책임 있는 커뮤니케이션 같은 말을 다 넣고 싶어진다.

그런데 그런 문장은 읽기에는 좋지만 행동을 크게 바꾸지 못한다. 에이전트가 실제로 틀리는 지점은 보통 더 구체적이다.

파일을 너무 넓게 고친다. 검증 없이 완료했다고 말한다. 초안을 공개 상태로 바꾼다. build 결과물을 직접 수정한다. 삭제나 push 앞에서 멈추지 않는다.

첫 규칙 파일은 이런 실패 모드만 잡아도 충분하다.

# Project Rules

- 요청받지 않은 파일은 수정하지 않는다.
- 새 글은 기본적으로 `draft: true`로 시작한다.
- build 산출물은 직접 수정하지 않는다.
- 삭제, push, deploy는 확인 전에는 하지 않는다.
- 완료 전 가능한 검증 명령을 실행한다.

이 정도면 화려하지는 않다. 대신 다음 작업에서 바로 쓸 수 있다.

내가 지금 쓰는 기준

지금은 프롬프트를 쓰기 전에 이 질문을 먼저 한다.

이 문장은 이번 요청에만 필요한가, 아니면 다음 작업에도 반복될 기준인가?

이번 요청에만 필요하면 프롬프트에 둔다. 다음 작업에도 반복될 기준이면 규칙 파일 후보로 본다. 틀렸을 때 되돌리기 어렵다면 규칙 파일만 믿지 않고 hook이나 권한 같은 차단 장치도 같이 본다.

비슷한 주제는 더 설명형으로 정돈할 수도 있다. 이 글은 그보다 내가 왜 반복 프롬프트를 규칙 파일 후보로 보기 시작했는지에 더 가깝다.

다음에 프롬프트를 길게 쓰고 있다면 바로 고치지 않아도 된다. 대신 세 번째 붙여넣는 문장만 따로 봐도 된다.

그 문장은 아마 프롬프트가 아니라 규칙 파일에 있을 자리다.

다음 글에서는 이 기준을 조금 더 넓혀, Karpathy가 말한 context engineering을 규칙 파일 관점에서 다시 읽어본다.
