Karpathy가 말한 컨텍스트 엔지니어링: 규칙 파일 다시 보기

Andrej Karpathy의 Software 2.0, Verifiability, MenuGen 회고를 렌즈로 삼아 AGENTS.md와 CLAUDE.md를 프롬프트 저장소가 아니라 작업 컨텍스트로 보는 기준을 정리한다.

이 글에서 다루는 것

  • 반복해서 붙여넣는 지시는 프롬프트가 아니라 에이전트가 다시 볼 컨텍스트로 옮겨야 한다.
  • Karpathy의 Software 2.0, Verifiability, MenuGen 회고를 AI agent 작업의 검증 기준으로 읽는다.
  • AGENTS.md와 CLAUDE.md에는 그럴듯한 원칙보다 멈춤 조건, 검증 조건, 반복 작업 기준을 남기는 편이 낫다.

앞 글에서는 같은 문장을 세 번째 붙여넣는 순간을 규칙 파일 신호로 봤다.

새 글은 draft: true로 시작한다. dist/는 건드리지 않는다. push나 deploy는 확인 전 하지 않는다. 완료 전 가능한 검증 명령을 실행한다.

이런 말은 매번 채팅창에 붙일 수 있다. 하지만 세션이 바뀌거나 문장 하나가 빠지면 기준도 같이 빠진다. 그래서 반복 지시는 프롬프트보다 repo 안의 컨텍스트로 옮기는 편이 낫다.

이 글은 같은 문제를 Andrej Karpathy의 글과 회고를 통해 다시 읽는다. 다만 Karpathy를 증명 도장처럼 쓰지는 않는다. 짧은 X 문구들은 접근이 막히거나 맥락이 잘리기 쉽다. 여기서는 Software 2.0, Verifiability, MenuGen 회고처럼 비교적 맥락을 따라갈 수 있는 글을 렌즈로 삼아, 내 repo 규칙을 어떻게 써야 하는지 본다.

반복 지시는 컨텍스트가 된다

프롬프트는 이번 요청의 목표를 담는 곳이다.

예를 들어 이런 문장은 단발성 프롬프트로도 쓸 수 있다.

이번 글은 초안까지만 만들어줘.
이 함수만 보고 버그 가능성을 확인해줘.
오늘은 코드 수정 없이 리뷰만 해줘.

이런 말은 그날의 요청이다. 다음 작업에서는 달라질 수 있다.

반대로 매번 반복되는 말이 있다.

- 새 글은 기본적으로 `draft: true`로 시작한다.
- 공개 발행, deploy, push는 별도 확인 후 진행한다.
- 완료 전 가능한 check/build 명령을 실행한다.

이건 그날의 부탁이 아니라 작업 환경의 기본값이다. AI에게 영어로 지시하는 일이 중요해질수록, 반복되는 영어 지시도 코드처럼 관리해야 한다. 어디에 두는지, 언제 읽히는지, 어떻게 검증되는지가 중요해진다.

그래서 AGENTS.md나 CLAUDE.md를 긴 프롬프트 저장소로만 보면 부족하다. 이 파일들은 에이전트가 작업할 때 다시 봐야 할 컨텍스트에 가깝다. 여기서 컨텍스트는 모델이 판단할 때 보는 파일, 명령, 금지 조건, 검증 기준을 뜻한다.

Software 2.0 렌즈로 보면 검증이 남는다

Karpathy의 Software 2.0 글에서 내가 가져온 핵심은 “사람이 모든 규칙을 직접 쓰지 않는 영역이 커진다”는 점이다. 신경망에서는 사람이 코드를 한 줄씩 쓰기보다 데이터, 모델 구조, 학습 과정이 프로그램의 큰 부분을 만든다.

AI 코딩도 비슷한 압력을 만든다. 내가 모든 줄을 직접 쓰지 않는다면, 내가 책임져야 할 것은 더 명확해진다.

  • 무엇을 만들지 설명하는 사양
  • 어떤 파일과 문서를 볼지 정하는 컨텍스트
  • 결과를 받아들일지 판단하는 검증 기준
  • 위험 작업 앞에서 멈추는 절차

Karpathy의 Verifiability 글은 이 지점을 더 직접적으로 건드린다. 자동화하기 쉬운 것은 내가 검증할 수 있는 일이다. 반대로 검증하지 못하는 일을 빠르게 맡기면, 속도는 올라가도 책임은 흐려진다.

그래서 규칙 파일에는 추상적인 가치보다 검증 가능한 문장이 더 잘 맞는다.

예를 들어 “좋은 코드를 작성한다”보다 “완료 전 npm run check와 npm run build를 실행한다”가 낫다. “사용자 경험을 고려한다”보다 “새 UI는 모바일과 데스크톱에서 텍스트 겹침을 확인한다”가 낫다. “안전하게 작업한다”보다 “삭제, push, deploy는 확인 전에는 진행하지 않는다”가 낫다.

좋은 규칙 파일은 에이전트를 똑똑하게 만드는 문서가 아니다. 검증 가능한 작업 단위를 더 자주 보게 만드는 문서다.

MenuGen 회고는 80% 착각을 보여준다

Karpathy가 MenuGen을 만들며 남긴 회고도 같은 방향으로 읽힌다. 첫 프로토타입은 금방 나온다. 자연어로 만들고, 실행 결과를 보며 고치면 작동하는 데모까지는 빠르다.

하지만 첫 결과를 보고 “80% done”처럼 느껴도, 실제로는 “closer to 20%”에 가까울 수 있다. 배포, 환경 변수, 인증, 결제, rate limit, deprecated API, dev/prod 설정으로 들어가면 일이 전혀 달라진다.

나는 이 지점이 규칙 파일 문제와 바로 이어진다고 본다. 감각 코딩(vibe coding)은 데모를 빨리 만든다. 하지만 서비스를 책임지는 작업으로 넘어가면 다음 질문이 필요하다.

어디까지 AI에게 맡길 것인가.
어떤 변경은 사람이 먼저 판단할 것인가.
무엇을 통과해야 완료라고 부를 것인가.
되돌리기 어려운 작업은 어디서 멈출 것인가.

이 질문의 답은 대화창보다 규칙 파일에 더 잘 남는다.

repo 규칙에는 실행 조건을 쓴다

Karpathy의 문장을 그대로 따라 쓰는 것이 목적은 아니다. 내가 가져갈 기준은 더 실무적이다.

규칙 파일에는 에이전트가 실제로 멈추거나 확인할 수 있는 문장을 둔다.

  • 위치 기준: 새 블로그 글은 src/content/blog에 둔다.
  • 기본 상태: 새 글은 기본적으로 draft: true로 시작한다.
  • 금지 범위: dist/ 같은 build 산출물은 직접 수정하지 않는다.
  • 승인 경계: 삭제, push, deploy는 확인 전에는 하지 않는다.
  • 검증 기준: 완료 전 npm run check와 npm run build를 실행한다.

이 문장들은 화려하지 않다. 대신 다음 작업에서 바로 작동한다. 에이전트가 코드를 쓰기 전에 봐야 할 것, 사람이 확인해야 할 것, 완료라고 부르기 전에 통과해야 할 것이 보인다.

반대로 “좋은 설계를 한다”, “사용자를 고려한다”, “안전하게 작업한다” 같은 문장은 그대로 두면 약하다. 필요한 말일 수는 있지만, 그것만으로는 행동이 바뀌기 어렵다. 이런 문장을 쓰려면 무엇을 보면 좋은 설계인지, 어떤 화면에서 사용자를 확인할지, 어떤 작업 앞에서 멈출지까지 같이 적어야 한다.

컨텍스트 엔지니어링은 긴 프롬프트가 아니다

컨텍스트 엔지니어링(context engineering)은 프롬프트를 길게 쓰는 일이 아니다. 모델이 다음 판단을 할 때 무엇을 보게 할지 설계하는 일에 가깝다.

여기서 중요한 단어는 “많이”가 아니라 “알맞게”다.

초보적인 실수는 모든 것을 한 파일에 넣는 것이다. 프로젝트 철학, 코드 스타일, 배포 절차, 보안 예외, 글쓰기 톤, 검수 기준, 장기 메모를 전부 root 규칙 파일에 넣으면 든든해 보인다. 하지만 실제 작업에서는 중요한 문장이 묻힌다.

컨텍스트 엔지니어링 관점에서 규칙 파일은 세 층으로 나누는 편이 낫다.

  • 최상위(root) 규칙은 모든 작업에 반복 적용되는 짧은 기준이다. 위험 작업 확인, 검증, 수정 범위처럼 매번 빠지면 안 되는 내용을 둔다.
  • 작업 분야(domain) 문서는 특정 작업 때만 필요한 기준이다. 글쓰기 가이드, UI 기준, 배포 절차처럼 작업 종류가 맞을 때만 꺼내 보는 쪽이 낫다.
  • 현재 프롬프트는 이번 요청의 목표와 예외다. “Karpathy 인용 중심 초안”이나 “파일로 저장”처럼 오늘 달라질 수 있는 조건을 둔다.

OpenAI의 Codex 문서는 AGENTS.md를 global, project, 현재 작업 경로 기준으로 instruction chain에 넣는다고 설명한다. 가까운 디렉터리의 지시가 더 구체적인 기준이 된다. Claude Code 쪽은 CLAUDE.md를 읽고, 이미 AGENTS.md를 쓰는 repo라면 import나 symlink로 연결할 수 있다고 안내한다.

도구별 로딩 방식은 다르다. 하지만 운영 관점의 결론은 같다.

반복 기준은 repo에 두고,
작업별 기준은 필요할 때만 보이게 하고,
이번 요청은 짧게 남긴다.
검증할 수 없는 변경은 작게 쪼개거나 멈춘다.

이게 내가 더 분명하게 잡은 기준이다. 프롬프트 엔지니어링은 모델에게 무엇을 말할지 다룬다. 컨텍스트 엔지니어링은 모델이 무엇을 보고 일하게 할지 다룬다. AI agent와 repo에서 일할 때는 후자가 점점 더 중요해진다.

결론

핵심 문장을 다시 쓰면 이렇다.

반복되는 기준은 프롬프트가 아니라 컨텍스트다.

Karpathy의 글과 회고를 렌즈로 삼아도 같은 곳에 도착한다. 사람이 모든 줄을 직접 쓰지 않는다면, 사람이 더 신경 써야 할 것은 사양, 컨텍스트, 검증, 멈춤 조건이다. 감각 코딩은 출발 속도를 올려줄 수 있지만, repo 규칙은 그 결과를 내가 책임질 수 있는 작업으로 바꾼다.

오늘 바로 적용할 기준은 이 정도면 충분하다.

  • 세 번 반복한 프롬프트 문장은 규칙 파일 후보로 옮긴다.
  • 규칙 파일에는 가치 판단보다 멈춤 조건과 검증 조건을 남긴다.
  • AI에게 바로 코드부터 요구하지 말고 접근법과 trade-off를 먼저 보게 한다.
  • 내가 검증할 수 없는 변경은 작게 쪼개거나 멈춘다.
  • 공개, 삭제, push, deploy처럼 되돌리기 어려운 일은 문서가 아니라 실제 확인 절차로 막는다.

좋은 프롬프트는 여전히 필요하다. 다만 좋은 프롬프트만으로는 반복 작업을 버티기 어렵다.

다음 글에서는 이 기준을 한 단계 더 좁혀, Codex Skill을 어떤 반복 절차에 써야 하는지 정리한다.
