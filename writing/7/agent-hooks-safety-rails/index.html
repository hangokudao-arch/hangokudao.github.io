에이전트 훅은 자동화가 아니라 사고방지 장치다

AI agent 작업에서 hook을 어디에 써야 하는지, AGENTS.md나 Skill과 어떻게 나눠야 하는지, 위험 작업을 막는 가드레일로 설계할 때의 기준을 정리한다.

이 글에서 다루는 것

  • Hook은 에이전트가 특정 이벤트를 지날 때 실행되는 검사 지점이며, 사고를 줄이는 가드레일에 가깝다.
  • AGENTS.md와 Skill은 에이전트가 참고하는 기준이고, hook은 도구 호출이나 작업 종료 같은 순간에 실제 검사를 붙인다.
  • 좋은 hook은 모든 것을 자동화하지 않고, 되돌리기 어려운 작업을 멈추거나 검토해야 할 정보를 드러낸다.

앞 글에서는 MCP를 AI에게 외부 도구와 자료를 연결하는 통로로 정리했다.

그다음에 헷갈리는 말이 hook이다.

처음에는 hook도 자동화처럼 보인다. 어떤 시점에 스크립트를 붙이고, 조건에 맞으면 무언가 실행한다. 그래서 “에이전트가 알아서 더 많이 하게 만드는 장치”처럼 느끼기 쉽다.

나는 지금 hook을 반대로 본다.

Hook은 에이전트에게 더 많은 일을 시키는 장치가 아니라, 위험한 순간에 멈추게 하는 사고방지 장치다.

이 글은 AI agent 작업에서 hook을 어디에 두면 좋은지 정리한다. 특히 AGENTS.md, Skill, permission, CI와 역할을 섞지 않고, “무엇을 막고, 무엇을 알리고, 무엇을 기록할지”로 나눠 본다.

훅은 특정 순간에 붙는 검사다

Hook은 말 그대로 어떤 흐름의 특정 지점에 걸어두는 동작이다.

Codex 기준으로 보면 hook은 에이전트 루프 안에 사용자가 만든 스크립트를 끼워 넣는 확장 지점이다. 예를 들어 사용자 프롬프트가 제출될 때, 도구 사용 전후, 승인 요청 직전, 대화가 끝나려는 시점, compact 전후 같은 이벤트에 붙을 수 있다.

중요한 점은 hook이 일반 지시문과 다르다는 것이다.

AGENTS.md는 에이전트가 읽고 따라야 할 repo 기준이다. “새 글은 draft: true로 시작한다”, “dist/는 직접 수정하지 않는다”, “push와 deploy는 확인 후 한다” 같은 기준이 여기에 맞다.

Skill은 특정 반복 작업의 절차다. 글 검수, git 변경 분리, UI 확인, 배포 준비처럼 “이 작업을 할 때는 이 순서로 본다”에 가깝다.

Hook은 더 기계적이다. 특정 이벤트가 발생했을 때 실제 스크립트를 실행한다. 예를 들어 Bash 명령을 실행하기 전에 명령 문자열을 검사하거나, 작업 종료 직전에 검증 명령을 빼먹었는지 확인하거나, prompt 안에 secret처럼 보이는 문자열이 들어왔는지 확인할 수 있다.

그래서 hook은 “기억해 줘”보다 “이 순간에는 확인해”에 가깝다.

사고는 보통 경계에서 난다

에이전트 작업에서 사고가 나는 지점은 대개 비슷하다.

파일을 읽는 것과 바꾸는 것의 경계. 로컬 commit과 remote push의 경계. 초안 작성과 공개 발행의 경계. 단순 명령 실행과 삭제, 권한 변경, 배포, production 접근의 경계.

프롬프트나 AGENTS.md는 이 경계를 설명할 수 있다. 하지만 설명만으로는 매번 충분하지 않다. 긴 세션에서 context가 길어지고, 사용자가 짧게 “그냥 배포해줘”라고 말하고, 에이전트가 앞 단계 검증을 이미 했다고 착각하면 사고가 난다.

이때 hook이 맡을 수 있는 역할은 명확하다.

  • 위험한 명령을 실행하기 전에 멈춘다.
  • 승인이 필요한 작업인지 다시 분류한다.
  • 작업이 끝나기 전에 빠진 검증을 알려준다.
  • 공개하면 안 되는 문자열이나 내부 경로를 발견하면 경고한다.
  • 에이전트가 보고한 “완료”와 실제 git 상태가 맞는지 확인한다.

여기서 핵심은 hook이 에이전트의 판단을 대체하지 않는다는 점이다. Hook은 판단해야 할 순간을 더 또렷하게 만든다. 사람이나 에이전트가 놓치기 쉬운 경계를 기계적으로 다시 보여준다.

어떤 훅이 사고를 줄이는가

내가 hook을 설계한다면 먼저 세 종류로 나눈다.

첫째, 실행 전 차단이다.

대표적으로 PreToolUse 같은 이벤트가 여기에 맞다. 에이전트가 Bash, 파일 수정, MCP tool 같은 도구를 쓰기 전에 입력을 검사한다. 삭제, 대량 overwrite, secret 파일 접근, public remote push처럼 위험한 패턴을 발견하면 실행 전에 막는다.

이 방식은 사고방지 효과가 크다. 이미 실행된 명령을 되돌리는 것보다 실행 전 멈추는 편이 낫기 때문이다.

둘째, 승인 요청 보강이다.

승인이 필요한 명령 앞에서 “이 요청은 왜 위험한가”를 더 잘 설명하게 만들 수 있다. 예를 들어 단순 build와 public deploy는 같은 shell 명령처럼 보여도 영향 범위가 다르다. Hook은 approval prompt가 올라오기 전에 작업 종류, 대상 remote, branch, 되돌림 경로를 검사할 수 있다.

셋째, 종료 전 검증이다.

작업이 끝났다고 말하기 전에 git status, git diff --check, test/build 결과, 공개 URL 확인 같은 항목이 빠졌는지 본다. 특히 에이전트는 “수정했다”와 “검증됐다”를 혼동할 수 있다. 종료 시점 hook은 이 둘을 나누는 데 도움이 된다.

이 세 가지를 한 문장으로 줄이면 이렇다.

실행 전에는 막고, 승인 전에는 설명하고, 종료 전에는 검증한다.

이 정도면 hook의 첫 목적은 충분하다.

훅은 완전한 보안 경계가 아니다

여기서 조심해야 할 점이 있다.

Hook을 만들었다고 모든 위험이 사라지는 것은 아니다. Codex 공식 문서도 PreToolUse를 가드레일로 설명하면서, 모든 작업 경로를 완전히 가로채는 강제 경계로 보지는 않는다. 지원되는 tool call을 검사할 수 있지만, 도구 종류나 실행 방식에 따라 빠지는 경로가 있을 수 있다.

PostToolUse도 마찬가지다. 도구 실행 뒤에 문제를 발견할 수는 있지만, 이미 일어난 side effect를 되돌리는 장치는 아니다. 삭제 명령이 이미 실행된 뒤라면 hook이 “문제 있음”을 알려도 파일이 자동으로 돌아오지는 않는다.

그래서 hook을 보안의 전부로 보면 위험하다.

더 강한 경계가 필요하면 permission, sandbox, branch protection, CI, protected environment, 리뷰 요구, 별도 배포 스크립트를 같이 봐야 한다. Hook은 그중 하나의 층이다.

나는 이렇게 나눠서 보는 편이 가장 안전하다고 본다.

  • AGENTS.md: repo에서 항상 지켜야 할 기준을 적는다.
  • Skill: 특정 반복 작업의 절차를 묶는다.
  • Hook: 도구 사용 전후, prompt 제출, 작업 종료 같은 이벤트에 검사를 붙인다.
  • Permission/sandbox: 에이전트가 할 수 있는 작업 범위를 제한한다.
  • CI/branch protection: remote에 들어가기 전 마지막 공개 경계를 둔다.
  • 배포 스크립트: 정해진 경로로만 배포하게 만든다.

Hook은 중요한 층이지만, 혼자서 모든 것을 막지는 않는다.

좋은 훅은 작고 설명 가능하다

Hook을 만들 때 가장 위험한 유혹은 “좋아 보이는 자동화”를 계속 붙이는 것이다.

예를 들어 작업이 끝날 때마다 자동으로 commit하고 push하고 deploy하게 만들 수도 있다. 하지만 이건 사고방지와 거리가 멀다. 오히려 되돌리기 어려운 작업을 사람의 확인 밖으로 밀어낼 수 있다.

또 하나 봐야 할 점은 hook 자체도 실행 코드라는 것이다. Codex는 관리되지 않는 command hook을 실행하기 전에 사용자가 현재 hook 정의를 검토하고 신뢰하도록 요구한다. 이 흐름은 귀찮은 절차가 아니라 안전 장치다. Hook이 사고를 막는다고 해도, 신뢰하지 않은 hook을 조용히 실행하면 그 hook 자체가 새로운 사고 지점이 된다.

좋은 hook은 작다.

예를 들어 이런 기준이 낫다.

  • rm -rf, git reset --hard, 대량 삭제, 권한 변경은 실행 전 멈춘다.
  • git push는 remote, branch, commit hash를 보여주고 별도 확인을 요구한다.
  • public deploy 전에는 source commit이 private remote에 반영됐는지 확인한다.
  • .env, key, token, private repo명처럼 보이는 문자열이 공개 글에 들어오면 경고한다.
  • 완료 보고 전에는 검증 명령을 실제로 실행했는지 확인한다.

이런 hook은 화려하지 않다. 대신 실패 모드가 분명하다. 무엇을 막는지, 왜 막는지, 사람이 어떻게 풀 수 있는지 설명할 수 있다.

반대로 나쁜 hook은 넓고 조용하다. 언제 실행되는지 모르고, 실패했을 때 이유가 불분명하고, 자동으로 상태를 바꾸며, 사람이 검토할 정보를 줄여버린다. 이런 hook은 안전 장치가 아니라 숨은 자동화가 된다.

처음에는 경고만으로도 충분하다

처음부터 강하게 막는 hook을 만들 필요는 없다.

처음에는 warning만 남겨도 얻는 것이 많다. 어떤 명령이 자주 위험 경계에 닿는지, 어떤 작업에서 검증을 빼먹는지, 어떤 문구가 공개 안전성 문제를 만드는지 볼 수 있기 때문이다.

내가 권하는 순서는 이렇다.

  1. 자주 반복되는 사고 후보를 적는다.
  2. 그 사고가 어느 이벤트에서 잡히는지 본다.
  3. 처음에는 경고로 시작한다.
  4. false positive를 줄인다.
  5. 되돌리기 어려운 작업만 차단으로 올린다.

예를 들어 “public repo에 source 파일이 올라가면 안 된다”는 사고 후보가 있다고 하자. 처음에는 deploy 전 tree에 금지 경로가 있는지 경고만 할 수 있다. 그다음 실제 배포 경로에서 통과해야 하는 검사로 올린다. 마지막으로 direct push까지 막아야 한다면 pre-push hook, branch protection, 배포 스크립트 검사를 함께 둔다.

이 순서가 중요한 이유는 간단하다. 너무 넓은 차단은 작업을 멈추게 만들고, 사람은 결국 우회 방법을 찾는다. 작게 시작해서 실제 실패 모드에 맞춘 hook만 남기는 편이 오래 간다.

내가 훅을 쓸 때의 기준

지금 내 기준은 이렇다.

  • 위험 작업 앞에 둔다. 삭제, overwrite, push, deploy, credential 접근처럼 되돌리기 어려운 순간이 우선이다.
  • 검증 가능한 조건만 둔다. “좋은 글인지 확인”보다 “공개 글에 내부 경로가 있는지 확인”이 hook에 맞다.
  • 자동 수정보다 중단과 설명을 우선한다. 특히 git, 배포, secret 관련 작업은 자동 해결보다 사람이 볼 정보를 주는 편이 낫다.
  • 실패 메시지는 사람이 행동할 수 있게 쓴다. “blocked”보다 “public remote push는 배포 스크립트를 사용하라”가 낫다.
  • Hook을 마지막 방어선으로 보지 않는다. permission, CI, branch protection, deploy script와 같이 둔다.

Hook은 에이전트를 더 자율적으로 만들기 위한 장치가 아니다. 오히려 자율성이 위험해지는 지점에서 속도를 낮추는 장치다.

그래서 나는 hook을 붙일 때마다 이렇게 묻는다.

이 hook이 없으면 어떤 사고가 날 수 있나?
그 사고는 실행 전, 승인 전, 종료 전 중 어디서 가장 잘 잡히나?
막아야 하나, 경고해야 하나, 기록만 하면 되나?
사람이 우회해야 할 때 이유와 대체 경로가 보이나?

이 질문에 답하지 못하면 아직 hook으로 만들 때가 아니다.

결론

에이전트 작업에서 hook은 편리한 자동화 장치처럼 보인다.

하지만 실제로 오래 쓰려면 사고방지 장치로 보는 편이 낫다. AGENTS.md가 기준을 남기고, Skill이 반복 절차를 묶고, permission과 CI가 강한 경계를 만든다면, hook은 그 사이의 특정 순간에서 “잠깐, 이거 확인했나?”라고 멈추게 한다.

좋은 hook은 많은 일을 하지 않는다. 위험한 일을 조용히 통과시키지 않는다. 사람에게 필요한 정보를 보여주고, 되돌리기 어려운 작업 앞에서 속도를 낮춘다.

AI agent를 실무 작업에 붙일수록 중요한 것은 더 빠른 자동화가 아니라 더 선명한 경계다. Hook은 그 경계를 기계적으로 다시 확인하게 만드는 작은 장치다.

다음에 hook을 만들 일이 생기면 기능 목록부터 쓰지 말고, 사고 목록부터 쓰는 편이 낫다.
